## 1. Goal

Magav is a Hebrew RTL volunteer-shift SMS reminder system that deploys to **two targets that must stay in lockstep**: a .NET 8 web backend + React frontend, and an Android Kotlin app (embedded Ktor server + Room/SQLCipher DB, native SMS). Introduce a **shift-type distinction** across the whole system: **משמרת מבצעית (operational)** = everything that exists today (Excel-imported and manually created shifts, all current schedulers); and **משמרת מנהלית (administrative)** = a NEW, always-manually-created shift type with its own creation page, its own fields, its own message templates, and its own SEPARATE reminder scheduler that fires at different times than the operational one. Operational behavior must remain functionally unchanged; administrative shifts must never bleed into operational queries/SMS and vice versa.

## 2. Detailed requirements (from the user)

- Rename the current shifts page to **"משמרות מבצעיות"** and add a NEW page **"משמרות מנהליות"**.
### Fields & page

- An administrative shift has fields: **description (תיאור)** — required; **date** — required; **time (שעה)** — required; **address (כתובת)** — **optional**; **location name (מיקום)** — **optional**; **vehicle (רכב)** — **OPTIONAL**; **vehicle location (מיקום רכב)** — **OPTIONAL**. (Vehicle and vehicle-location are optional because the user does not always know in advance which vehicle will be available.) **Multiple volunteers** can be assigned to one shift.
- **Per-week view (DECIDED):** the administrative page looks like the operational page (it shows the shift **description**) but lists a whole **week, Sunday→Saturday**, with controls to move to the previous/next week (instead of the operational per-day navigation).

### Save + send SMS on creation (DECIDED)

- On save, the user can either **(a) just save**, or **(b) save AND send SMS** to the assigned volunteers.
- **The assignment SMS uses a template chosen ONCE as a system-wide setting** (the user selects one of the message templates they authored) — it is NOT typed per shift. The plan must decide WHERE this one-time selection lives (recommendation in Section 4g). User's example assignment template (note: no location/address line — those fields are optional):
  ```
  שלום {שם},
  שובצת למשימה {תיאור},
  בתאריך {תאריך} שעה {שעה}
  בהצלחה
  ```

### Manual per-volunteer "send SMS" icon (DECIDED)

- Next to each assigned volunteer, show a **send-SMS icon** the user can click to message that single volunteer. **This mirrors an affordance that ALREADY EXISTS** in the operational UI — the per-row `MessageSquare` icon in `ShiftsManagementPage.tsx` (~lines 789-804) that calls `shiftsService.sendShiftSms` → `POST /api/shifts/{id}/send-sms` (auto-selects template by date, no dedup, always sends).
- When the shift is **today**, this icon sends a dedicated **"your shift is today" message** (a NEW admin template, also selected once in settings). See Section 4f for the not-today behavior, which is an open question.

### Administrative reminder scheduler (DECIDED — ONE automated reminder only)

- A new settings page **"הגדרות תזמון למשמרות מנהליות"**, a **SEPARATE scheduler** from the operational one, sending at different times.
- **Only an ADVANCE reminder is automated. There is NO automated same-day reminder** for admin shifts (they generally run early morning; the same-day notification is handled manually via the per-volunteer icon above).
- The advance reminder is sent **one day before the shift** at a **user-configurable hour**, and **must be Shabbat- AND holiday-aware: a Saturday OR a Jewish holiday counts as a non-working day, and the send rolls back to the previous working day.** So a **Sunday** shift's advance reminder goes out **Friday**, and shifts whose day-before falls on a holiday roll back accordingly. This is exactly what the existing **`WeekdayAdvance`** reminder type + the holiday-aware DayGroup logic already do (a Jewish holiday is mapped to the `Sat` DayGroup) — reuse that machinery with `DaysBeforeShift = 1` (Section 4c).

## 4. Key design decisions the plan MUST resolve

Pose each as an explicit decision with trade-offs, give a recommendation **with reasoning**, but present it so the user/implementer can confirm. Do not silently pre-decide.

**(a) Shift-type discriminator — column vs separate table.**
- Option A: add `ShiftType TEXT` (e.g. `'Operational'|'Administrative'`, default `'Operational'`) to `Shifts`/`ShiftEntity`, plus the new admin-only columns (`Description`, `Time`, `Address`, `VehicleLocation`, …) as nullable. Pros: reuse repositories/DAOs/dedup; one code path. Cons: every existing active-shift query must add a type filter or admin shifts leak into operational views/SMS; schema bloat with many nullable columns; differing required-field semantics.
- Option B: a separate `AdministrativeShifts` table (+ its own DAO/repo, and decide how its volunteer assignment works). Pros: clean isolation, no cross-talk risk, independent lifecycle. Cons: parallel code paths on both platforms, duplicated dedup/scheduler plumbing.
- Either way: **existing rows backfill/default to operational.** State the exact backfill mechanism per platform.
- **DECIDED — multiple volunteers per admin shift, using the SAME pattern as operational.** Admin shifts reuse the existing **one-row-per-volunteer** model (N volunteers = N `Shifts` rows sharing one shift identity) — do NOT introduce a junction table. This keeps `SmsLog` dedup `(ShiftId, ReminderType)` working per-volunteer with zero change.
- **Group key — user chose `(Date, Time, Description)`.** This ties an admin shift's rows together (operational uses the implicit `(ShiftDate, ShiftName, CarId)` tuple; note CarId is dropped here because vehicle is optional). The plan should ADOPT this but explicitly state the trade-offs and the handling required, since it is a natural/composite key:
  - **Acceptable and consistent** with how operational already uses a natural composite key — recommended to proceed.
  - **Caveat 1 — editing re-buckets:** because Description/Time are part of the key, the group create/update endpoint MUST update all rows of a group atomically; editing the Description of a saved shift effectively moves its rows to a new group key.
  - **Caveat 2 — collisions:** two genuinely-different admin shifts with the identical `(Date, Time, Description)` would merge into one group (rare, since Description is free text, but state it). The `Time` value must be normalized (e.g. zero-padded `HH:mm`) so `"8:00"` and `"08:00"` don't split a group.
  - **Alternative to mention, not adopt:** a dedicated immutable `ShiftGroupId` column avoids both caveats (robust to description edits and allows duplicate identical shifts) at the cost of one extra column + generating/threading the id. Recommend `(Date, Time, Description)` per the user's choice unless the planner finds the edit/collision caveats unacceptable, in which case flag `ShiftGroupId` for user confirmation.
  - This same group key drives the per-week listing grouping (4e) and the group create/update/cancel/send endpoints.

**(b) Query isolation (non-negotiable correctness).** Enumerate EVERY query/code path that lists/sends "active" shifts and specify how each gets a shift-type filter so the operational scheduler NEVER sends to administrative shifts and vice versa. This must explicitly cover: `SmsReminderService.cs` eligibility query, `ShiftsRepository.cs` active queries, the .NET shift endpoints in `Program.cs`, the Android `SmsReminderService.kt`/`ShiftDao.kt` queries, and the Ktor `ShiftRoutes.kt` endpoints. Treat a missing filter as a P0 defect.

**(c) Administrative scheduler design.** Administrative shifts are **manual/ad-hoc**, not Excel-bulk and not aligned to the Sun-Thu/Fri/Sat patrol rhythm. The **trigger RULES are DECIDED** (below) — the plan must design the mechanism that implements them, NOT re-choose them:

- **ONE automated reminder only: the ADVANCE reminder.** It is sent **one day before the shift** at a **user-configurable hour**. **NO automated same-day reminder exists** for admin shifts (per the user, they generally run early morning; same-day notification is manual via the per-volunteer icon — Section 4f).
- **Holiday- & Shabbat-aware rollback:** a **Saturday OR a Jewish holiday** is a non-working day; the advance send rolls back to the previous working day. A **Sunday** shift → reminder on **Friday**; a shift whose day-before is a holiday rolls back to the prior working day too.
- **This is exactly the existing `WeekdayAdvance` + holiday machinery — REUSE it (don't reinvent).** Verified facts the plan can rely on:
  - `GetEffectiveDayGroupAsync` (`SmsSchedulerService.cs` ~149-166) and `effectiveDayGroupForDate` (`SmsSchedulerWorker.kt` ~230-236) already map **a Jewish holiday → `Sat` DayGroup**, and "tomorrow is a holiday" → `Fri`.
  - Holidays live in the `JewishHolidays` table (web) / `JewishHolidayDao` (Android), seeded ~2025-2035 idempotently (`INSERT OR IGNORE`); query is `IsHolidayAsync(dateString)`.
  - `WeekdayAdvance` already computes `windowEnd = NextWorkingDay(today) + N` with `runLogTargetDate = today` (the firing day), and `BuildMessage` derives `{תאריך}`/`{יום}` from **each shift's own date**, not the firing day. Setting `DaysBeforeShift = 1` reproduces "1 working-day before."
  - Heed the documented **WeekdayAdvance window/runlog coupling** gotcha (per-shift message date; single `TargetDate` param shared by the shift query and the runlog dedup key).
- The advance reminder's message uses a template configured on the new "הגדרות תזמון למשמרות מנהליות" page (hour + enabled + template).

## 5. Hard constraints the plan MUST honor (non-negotiable)

- **Both targets in sync.** Every model/constant/scheduler/seed change is a first-class deliverable on BOTH .NET and Android. Constants in `MagavConstants.cs` and `Constants.kt` must carry identical string values. The Android changes are NOT an afterthought.
- **Room migration discipline (silent data-loss risk).** Per CLAUDE.md: ANY change to a Room `@Entity` (columns/indices/FKs/PK/`defaultValue`/`@ColumnInfo`) requires, every time: (1) bump `@Database(version = 8 → 9)` in `MagavDatabase.kt`; (2) add an idempotent `MIGRATION_8_9` (use `ALTER TABLE … ADD COLUMN`, `CREATE INDEX IF NOT EXISTS`); (3) register it in BOTH `.addMigrations(...)` sites in `MagavApplication.kt`; (4) ensure the migration produces a schema that EXACTLY matches the entity (every index) so the schema-hash check passes. `fallbackToDestructiveMigration` is intentionally NOT used and must stay that way; a mismatch must crash visibly, not wipe the DB. If administrative shifts require a second new table/entity, repeat the full cycle for it.
- **Scheduler-config seeding gotcha.** Fresh-DB seed methods (`SeedSchedulerConfigAsync`, `SeedMessageTemplatesAsync`, `DatabaseInitializer.seedSchedulerConfigs()`) run ONLY on fresh DBs. New admin configs/templates must be hardcoded in the listed seed sites AND have an **idempotent migration/backfill path** (INSERT OR IGNORE style) so EXISTING production DBs (web and Android) receive them on next startup. Per project memory: keep the relevant defaults in sync across the (3) hardcoded places. Account for MessageTemplate FK integrity (a config's `MessageTemplateId` must reference a template that actually exists in upgraded DBs).
- **Security.** Parameterized queries only (`@0,@1,…` / Room params) — never concatenate SQL. Every new .NET endpoint `.RequireAuthorization()`; every new Ktor route `authenticate("auth-bearer")` unless intentionally public. Generic Hebrew error messages only; use `Results.Json()` with `ApiResponse<T>`, never `Results.Problem()`; no exception details/stack traces/column names in responses.
