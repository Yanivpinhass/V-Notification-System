# Domain Model — Magav V-Notification-System

Magav is a Hebrew RTL volunteer-shift management + SMS-reminder system. This document is the system-wide synthesis of the domain: the ubiquitous language, the bounded contexts that own each concept, the cross-component domain rules, and concept ownership. The single most important structural fact for everything below: **the same domain model is implemented THREE times** — the canonical .NET `Magav.Common.Models` POCOs, the Android Room `@Entity` classes, and the React TypeScript service DTOs — and these three are kept in sync **manually**, with no shared schema or codegen. Drift between them is the dominant risk in the system.

## 2. Bounded Contexts

The system has one logical domain but **two independent runtime implementations of it** plus a shared UI projection. There is no shared persistence and no compile-time edge between the React/Android clients and the .NET backend (discovery §4).

| Context | Owns / implements | Implementation | Notes |
|---|---|---|---|
| **Web backend (canonical)** | The authoritative domain model + business logic. Entities (`Magav.Common.Models`), repositories + services (`Magav.Server`), HTTP surface (`Magav.Api`). | .NET 8, SQLCipher SQLite at `db/magav.db`. | The canonical source of truth for entity shape, constants (`MagavConstants`), and business rules. [HIGH] |
| **Android backend (independent mirror)** | A full re-implementation of the SAME domain — Room entities, DAOs, Ktor routes mirroring the .NET API, native SMS scheduler. | Kotlin, Room + SQLCipher, on-device. | A **separate, standalone database and codebase**. It does not call the .NET backend; it re-implements it. Must mirror `MagavConstants` exactly (DR-009). | **Shared UI (React SPA)** | The Hebrew RTL admin UI + TS DTOs. The SAME built bundle is served by .NET/Nginx AND embedded in the Android WebView. | React 18 + TS, talks HTTP to whichever backend is hosting it (`localhost:5015`). | Defines its own TS DTO mirror of the entities. **Same term, different meaning / shape — flagged ambiguities:**

- **Role vs Roles.** The backend domain stores a single `Role` string on `User`; the client domain (auth payload + `lib/auth.ts`) treats it as `roles: string[]`. The `AuthService` wraps `Role → [Role]` at the boundary (DR-007). Additionally, the **users-list DTO** (`UserDto`) exposes singular `role`, while the **auth/session DTO** (`UserInfo`) exposes plural `roles[]` — two different shapes for the same concept in the same client (`usersService.ts:3-13` vs `authService.ts:16-21`). [HIGH]
- **Reminder "DayGroup".** A `DayGroup` value (`SunThu`/`Fri`/`Sat`) on a `SchedulerConfig` means *the day the SMS is sent*, not the day of the shift it is about (DR-002). This is the system's single most invertible concept — every component doc independently flags it.
- **Refresh-token lifetime.** Same concept, **different value** per backend: .NET default = **7 days** (`AuthService.cs` / `appsettings` `RefreshTokenExpirationDays`); Android = **3 days** (`JwtConfig.kt:16-17`). A session valid on web may be expired on Android. [HIGH]
- **Seeded admin credentials.** Same "default admin" concept, **different credentials**: .NET seeds `admin`/`Admin123!` (BCrypt cost 12); Android seeds `Admin`/`12345` (BCrypt cost 10). [HIGH]
- **`IsEnabled` / boolean-as-int.** SQLite has no bool; booleans (`IsEnabled`, `IsCanceled`, `IsActive`, `ApproveToReceiveSms`, `MustChangePassword`) are `int` 1/0 across all three platforms. The client TS types `isEnabled` as `number` (`schedulerService.ts`). [HIGH]
- **`Date`-as-string.** `JewishHoliday.Date` and `SchedulerRunLog.TargetDate`/`RanAt` are stored as **strings**, not DateTime, to avoid cross-provider format ambiguity. Holiday matching is string `yyyy-MM-dd` comparison. [HIGH]
- **`AiShiftVolunteerDto` (Android only).** A query DTO referencing a `getShiftsWithVolunteers()` JOIN that does not exist — dead/orphan, with no .NET counterpart (`db/dao/AiQueryDtos.kt`). [HIGH]
- **`CallbackConfig` + `GET/PUT /api/callback-config` (Android only).** A real domain entity and REST contract that exist in the Android Ktor surface ONLY — there is NO .NET `Magav.Api` counterpart and NO web-served implementation; the React `CallbackSettingsPage` is gated to the Android WebView (renders a Hebrew "Android-app-only" notice when `window.NativeMedia` is absent). This is the SAME accepted "Android-only native setting" pattern as the SIM-selection endpoint `GET/PUT /api/settings/sms-sim` (`SettingsRoutes.kt:54,68`) — **accepted by design, NOT a new defect** (ISS-004's accepted-divergence list). [HIGH]

### Summary
- **Domain rules captured:** 21 (DR-001–021; DR-020/021 are Android-native-only). **Bounded contexts:** 3 (web backend canonical, Android independent mirror, shared React UI). **Core concepts with explicit ownership:** 12 (incl. the Android-only `CallbackConfig`).
- **Most important cross-component fact:** the same domain is implemented three times and hand-synced (DR-001); the most invertible concept is DayGroup = send-day not shift-day (DR-002); the most subtle live divergences are refresh-token lifetime (7d web vs 3d Android) and seeded admin credentials. **Android-native-only carve-outs** (License/biometric/SIM, and now Auto-Callback-to-Gate / `CallbackConfig`, DR-020/021) deliberately have no .NET canonical — the same accepted pattern as `/api/settings/sms-sim` (ISS-004 accepted-divergence list), NOT drift.
